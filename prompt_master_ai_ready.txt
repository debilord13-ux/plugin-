==================================================
5. МЕТАДАННЫЕ ПЛАГИНА
==================================================

Metadata парсятся через AST.
Парсер читает верхнеуровневые присваивания вида:
- `__id__ = "..."` 
- `__name__ = "..."` 
- и так далее

Надёжный набор metadata:

```py
__id__ = "my_plugin"
__name__ = "My Plugin"
__description__ = "Short description."
__version__ = "1.0.0"
__author__ = "@username"
__icon__ = "packname/0"
__min_version__ = "11.12.0"
```

Подтверждённые поля PluginMetadata:
- id
- name
- description
- author
- version
- icon
- min_version

Правила metadata:
- только простые строковые литералы
- без `+`
- без `format`
- без `f"..."` 
- без вычислений
- без вызовов функций

Плохо:

```py
__id__ = "my_" + "plugin"
__name__ = f"Plugin {1}"
```

Хорошо:

```py
__id__ = "my_plugin"
__name__ = "Plugin"
```

Минимум, который нужно задавать всегда:
- `__id__`
- `__name__`

Практически безопасно всегда задавать весь блок metadata полностью.

9. HookStrategy и HookResult
==================================================

Подтверждённые HookStrategy:
- `HookStrategy.DEFAULT`
- `HookStrategy.CANCEL`
- `HookStrategy.MODIFY`
- `HookStrategy.MODIFY_FINAL`

Подтверждённые поля HookResult:
- `strategy`
- `request`
- `response`
- `update`
- `updates`
- `error`
- `params`

Правило:
- если ничего не меняешь, возвращай `HookResult()`
- если хочешь отменить действие, возвращай `HookResult(strategy=HookStrategy.CANCEL)`
- если меняешь request / response / update / params, клади объект в правильное поле

Пример send hook:

```py
from base_plugin import BasePlugin, HookResult, HookStrategy


class Plugin(BasePlugin):
    def on_plugin_load(self):
        self.add_on_send_message_hook()

    def on_send_message_hook(self, account, params):
        if isinstance(getattr(params, "message", None), str) and params.message == ".hi":
            params.message = "Hello"
            return HookResult(strategy=HookStrategy.MODIFY, params=params)
        return HookResult()
```

Практические правила:
- message send hook регистрируй через `add_on_send_message_hook()`
- request/update hooks регистрируй через `add_hook(...)`
- не забывай возвращать `HookResult`

12. XPOSED HOOKS
==================================================

Подтверждены базовые классы:
- `XposedHook`
- `MethodHook`
- `MethodReplacement`

Подтверждённые методы:
- `MethodHook.before_hooked_method(self, param)`
- `MethodHook.after_hooked_method(self, param)`
- `MethodReplacement.replace_hooked_method(self, param)`

Надёжная схема:
- находишь Java method через reflection
- вызываешь `self.hook_method(...)`
- при необходимости сохраняешь unhook-объект

Пример:

```py
from base_plugin import BasePlugin, MethodHook
from hook_utils import find_class


class TitleHook(MethodHook):
    def __init__(self, plugin):
        self.plugin = plugin

    def before_hooked_method(self, param):
        self.plugin.log("method called")


class Plugin(BasePlugin):
    def on_plugin_load(self):
        action_bar = find_class("org.telegram.ui.ActionBar.ActionBar")
        char_seq = find_class("java.lang.CharSequence")
        if not action_bar or not char_seq:
            return
        try:
            method = action_bar.getDeclaredMethod("setTitle", char_seq)
            self.hook_method(method, TitleHook(self))
        except Exception as e:
            self.log(str(e))
```

Практические правила:
- reflection ломкая, всегда `try/except`
- если поле/метод по имени не находится, ищи выше по class hierarchy
- safe mode может ломать или ограничивать hook behavior

27.1 Простая настройка + send hook
--------------------------------------------------

```py
__id__ = "caps_guard"
__name__ = "Caps Guard"
__description__ = "Edits outgoing text."
__version__ = "1.0.0"
__author__ = "@username"
__min_version__ = "11.12.0"

from base_plugin import BasePlugin, HookResult, HookStrategy
from ui.settings import Switch


class Plugin(BasePlugin):
    def on_plugin_load(self):
        self.add_on_send_message_hook()

    def create_settings(self):
        return [
            Switch(key="enabled", text="Enable", default=True),
        ]

    def on_send_message_hook(self, account, params):
        if not self.get_setting("enabled", True):
            return HookResult()
        text = getattr(params, "message", None)
        if isinstance(text, str) and text.isupper():
            params.message = text.lower()
            return HookResult(strategy=HookStrategy.MODIFY, params=params)
        return HookResult()
```

29. ЧЕКЛИСТ ПЕРЕД ВЫДАЧЕЙ ПЛАГИНА
==================================================

Перед тем как отдать код, проверь:

- Есть `__id__`
- Есть `__name__`
- Metadata статические
- Есть `class Plugin(BasePlugin)`
- Импорты только из реального API
- Нет тяжёлых действий в UI thread
- Все hooks возвращают `HookResult`
- Reflection обёрнут в `try/except`
- `fragment/activity` проверяются на `None`
- Настройки строятся через `ui.settings`
- Menu items строятся через `MenuItemData`
- Если меняется message, то меняется `params.message`
- Если есть markdown entities, они конвертируются в TLRPC entities

30. КАК ИИ ДОЛЖНА ОТВЕЧАТЬ ПОЛЬЗОВАТЕЛЮ
==================================================

Когда пользователь просит плагин:
- не объясняй долго
- не давай куски
- не пиши “вот пример”
- сразу давай полный рабочий код
- если нужен `.plugin`, возвращай содержимое готового `.plugin`

Если задача рискованная:
- выбери более простой, но реальный вариант
- не выдумывай unavailable API

Если пользователь просит “крутой” плагин:
- добавляй настройки
- menu item
- bulletins
- обработку ошибок
- аккуратный fallback
- не забывай производительность

==================================================
31. КОРОТКАЯ ФОРМА ДЛЯ ДРУГОЙ ИИ
==================================================

Если нужно совсем коротко задать роль другой ИИ, можно использовать это:

Ты пишешь только рабочие ExteraGram plugins в виде одного полного `.plugin` файла.
Используй только реальные модули ExteraGram: base_plugin, client_utils, android_utils, hook_utils, file_utils, markdown_utils, ui.alert, ui.bulletin, ui.settings.
Metadata пиши только статическими строками через top-level `__id__`, `__name__`, `__description__`, `__version__`, `__author__`, `__min_version__`, `__icon__`.
Плагин всегда делай через `class Plugin(BasePlugin)`.
Для send hook используй `add_on_send_message_hook()` и `on_send_message_hook(self, account, params)`.
Для request/update hooks используй `add_hook(...)`, `pre_request_hook`, `post_request_hook`, `on_update_hook`, `on_updates_hook`.
Для menu items используй `MenuItemData` и `MenuItemType`.
Для настроек используй только `Switch`, `Selector`, `Input`, `Text`, `Header`, `Divider`, `EditText`.
UI делай через `run_on_ui_thread`, тяжёлую работу через `run_on_queue`.
Если меняешь flow, возвращай `HookResult` с правильным `HookStrategy`.
Ничего не выдумывай, не используй внешние зависимости, не делай metadata динамическими.

33.19 Уровни надёжности этой инструкции
--------------------------------------------------

Уровень A. Подтверждено кодом:
- всё, что извлечено из `app.imy`
- всё, что извлечено из `build.json`
- всё, что извлечено из dex strings

Уровень B. Подтверждено кодом и согласуется с документацией:
- использование `BasePlugin`
- send hooks
- menu items
- settings controls
- alert/bulletin helpers
- markdown entities

Уровень C. Есть в документации, но не всегда напрямую подтверждено текущим pyc:
- полный список возможных context keys в menu callback
- отдельные сигнатуры некоторых Java-side Telegram методов
- некоторые экзотические TL object names для hooks

Правило:
- если что-то уровня C, всегда предпочитай более простой вариант уровня A/B

==================================================
34. ДОКАЗАТЕЛЬНЫЙ CHECKLIST
==================================================

Перед тем как доверять коду плагина, спроси:

1. Использует ли он только модули, реально найденные в `app.imy`?
2. Совпадают ли сигнатуры с тем, что подтверждено в `base_plugin.pyc`, `client_utils.pyc`, `ui/*.pyc`?
3. Не использует ли он dynamic metadata, хотя metadata parser ждёт constants?
4. Не придуман ли у него новый тип hook result или новый settings control?
5. Не импортирует ли он `run_on_queue` из неправильного модуля?
6. Не строится ли он на сомнительном имени internal engine вместо `BasePlugin` API?

Если ответ плохой хотя бы на один из пунктов, код надо упрощать до подтверждённого API.

### BasePlugin.hook_method / hook_all_methods / hook_all_constructors

```py
def hook_method(self, method_or_constructor, xposed_hook, priority=0):
    if xposed_hook is None:
        raise TypeError("xposed_hook cannot be None")
    if ExteraConfig.pluginsSafeMode:
        return None
    java_cb = self.__get_java_callback_instance(xposed_hook, priority)
    unhook = XposedBridge.hookMethod(method_or_constructor, java_cb)
    PluginsController.getInstance().addXposedHook(self.id, unhook)
    return unhook
```

```py
def hook_all_methods(self, hook_class, method_name, xposed_hook, priority=0):
    if xposed_hook is None or method_name is None:
        raise TypeError("xposed_hook and method_name cannot be None")
    if ExteraConfig.pluginsSafeMode:
        return []
    unhooks = XposedBridge.hookAllMethods(hook_class, method_name, self.__get_java_callback_instance(xposed_hook, priority))
    PluginsController.getInstance().addXposedHooks(self.id, list(unhooks))
    return unhooks
```

```py
def hook_all_constructors(self, hook_class, xposed_hook, priority=0):
    if xposed_hook is None:
        raise TypeError("xposed_hook cannot be None")
    if ExteraConfig.pluginsSafeMode:
        return []
    unhooks = XposedBridge.hookAllConstructors(hook_class, self.__get_java_callback_instance(xposed_hook, priority))
    PluginsController.getInstance().addXposedHooks(self.id, list(unhooks))
    return unhooks
```

Что это доказывает:
- Xposed-hooks реально поддерживаются из Python side
- safe mode может их блокировать
- unhook objects регистрируются в `PluginsController`

Главный practical pattern:
- перехватить `.command`
- запустить фоновой worker/thread
- показать loading UI
- отменить оригинальную отправку через `HookResult(strategy=HookStrategy.CANCEL)`

Живой пример из `downloader.plugin`:

```py
def on_send_message_hook(self, account, params):
    if not hasattr(params, "message") or not isinstance(params.message, str):
        return HookResult()

    msg = params.message.strip()

    if msg.startswith(".down ") or msg.startswith(".dl "):
        if self._task_in_progress.locked():
            BulletinHelper.show_error("Another download is already running")
            return HookResult(strategy=HookStrategy.CANCEL)

        thread = threading.Thread(
            target=self._process_download_and_send,
            args=(url, dialog_id, notify, schedule_date, mode_override, params.replyToMsg, params.replyToTopMsg),
            daemon=True
        )
        thread.start()

        run_on_ui_thread(lambda: self._show_loading_alert())
        return HookResult(strategy=HookStrategy.CANCEL)

    return HookResult()
```

Ещё один живой pattern из `tailed-tts.plugin`:

```py
def on_send_message_hook(self, account, params):
    if not isinstance(params.message, str) or not params.message.startswith(".tts"):
        return HookResult()

    text = ' '.join(params.message.strip().split()[1:])
    if len(text) == 0:
        BulletinHelper.show_info("usage...")
        return HookResult(strategy=HookStrategy.CANCEL, params=params)

    self.params = params
    threading.Thread(target=self.handle_tts, daemon=True).start()
    return HookResult(strategy=HookStrategy.CANCEL, params=params)
```

Очень важное замечание:
- в corpus встречаются плохие type hints
- например, один plugin может аннотировать `on_send_message_hook(...) -> HookStrategy`, но реально возвращать `HookResult`
- в таких случаях доверяй runtime spec, а не аннотации автора plugin-а

В новом corpus дополнительно замечены ловушки:

### `remove_on_send_message_hook()`

Такой вызов встречается в некоторых plugins, например в `TextTransformerRN`.

Правило:
- не используй `remove_on_send_message_hook()` как verified API
- если нужно снять send hook, полагайся на verified hooks lifecycle или перепроверяй runtime отдельно

### Некачественные type annotations

В corpus всё ещё встречаются случаи, где:
- `on_send_message_hook(...) -> HookStrategy`
- но реально возвращается `HookResult`

Правило:
- всегда доверяй runtime spec, а не случайным type hints из community files

38.4 Request hooks are no longer edge-case patterns
--------------------------------------------------

В текущем наборе `pre_request_hook` и `post_request_hook` уже реально встречаются как рабочие механизмы, а не как редкая экзотика.

Живой current example:

```py
def pre_request_hook(self, request_name: str, account: int, request: Any) -> HookResult:
    if not self.get_setting(self.SETTINGS_ENABLED, True):
        return HookResult()

    changed = self._apply_rules(request_name, request)
    self._record_entry("OUT", request_name, request, account=account, changed=changed)

    if changed:
        return HookResult(strategy=HookStrategy.MODIFY, request=request)
    return HookResult()
```

И post-request pattern:

```py
def post_request_hook(self, request_name: str, account: int, response: Any, error: Any) -> HookResult:
    ...
    return HookResult()
```

Практический вывод:
- если задача связана с outbound Telegram requests, `pre_request_hook` это уже нормальный design choice
- если нужен inspection/logging/packet studio behavior, можно строиться на request hooks, а не только на send hooks
- если request меняется, корректная форма возврата: `HookResult(strategy=HookStrategy.MODIFY, request=request)`

Observed add_hook names в текущем наборе:

```text
TL_messages_forwardMessages
TL_messages_sendMedia
updateStatus
on_message_hook
```

Важно:
- `TL_messages_forwardMessages` в этом наборе встречается особенно часто, но это partly family-biased из `UiTweaks`
- `on_message_hook` не подтверждён runtime spec и должен считаться подозрительным community-specific pattern

40. SAFE SOURCE-DERIVED HOOK RECIPES

This section converts source understanding into safer plugin design recipes.

--------------------------------------------------
40.1 Recipe: attach custom UI to a Telegram screen
--------------------------------------------------

Use this mental model:

1. Hook screen `createView`
2. Get `fragmentView`
3. Try to find native anchor
4. If anchor is unstable, add overlay to root
5. Cleanup in `onFragmentDestroy`

Why this is strong:
- matches upstream screen lifecycle
- avoids random timing bugs
- easy to refresh in `onResume`

--------------------------------------------------
40.2 Recipe: add a profile button or profile block
--------------------------------------------------

Best practice:
- source map to `ProfileActivity`
- try anchor around known profile content fields
- if exact field missing, attach to `fragmentView` or its parent
- avoid mutating `RecyclerView` during layout if overlay can solve the problem

Practical warning:
- direct insertion into a list while it is computing layout can crash
- if you are not 100% sure about the timing, use delayed attach / post / root overlay

--------------------------------------------------
40.3 Recipe: add a backdrop or hero image to profile/chat
--------------------------------------------------

Best practice:
- use a plugin-owned layer
- keep image logic separated from native avatar view
- if native screen host is unstable, attach backdrop to higher parent or `decorView`
- use gradient scrim to preserve readability

Do not:
- rely only on margins if the host itself is wrong
- confuse avatar presentation with backdrop presentation

--------------------------------------------------
40.4 Recipe: intercept sending or sending-adjacent behavior
--------------------------------------------------

Best practice:
- first ask: does verified ExteraGram helper already exist?
- if yes, use helper
- only read `SendMessagesHelper` source to understand semantics and naming

- file send

- safer document handling

40.5 Recipe: build Telegram-looking settings or tool overlays
--------------------------------------------------

Best practice:
- use `Theme.getColor`
- use rounded `GradientDrawable`
- use Telegram density via `AndroidUtilities.dp`
- if plugin settings screen is unstable, build in-plugin overlay settings

This pattern is especially strong for:
- file editors
- proxy dashboards
- profile tool panels
- visual UI plugins

--------------------------------------------------
40.6 Recipe: image injection without layout disasters
--------------------------------------------------

If the plugin injects big images:
- use dedicated container
- keep image separate from content cards
- use scrim/fade
- avoid changing content view paddings unless necessary
- test both `fragmentView` host and higher parent host

42. FINAL AI INSTRUCTION FOR SOURCE-AWARE GENERATION
==================================================

When official Telegram source is relevant, the AI must do all of the following:

1. Identify the likely upstream screen/component.
2. Use upstream code only as architectural guidance.
3. Verify every practical assumption against ExteraGram runtime evidence.
4. Prefer verified helpers over internal reflection when both can solve the task.
5. Use lifecycle-safe attach/detach patterns.
6. Prefer stable overlays over brittle “perfect native injection”.
7. If a source-derived hypothesis fails locally, degrade safely and keep the plugin loadable.

The best source-aware ExteraGram plugin is not the one that copies Telegram internals most aggressively.
It is the one that:
- understands Telegram’s structure,
- verifies it against ExteraGram,
- and then chooses the most stable implementation path.

43. FIELD LESSONS FROM REAL PLUGIN ITERATIONS
==================================================

This section is not abstract theory.
It captures failure patterns and working patterns discovered while iterating on many actual ExteraGram plugins.

This section should strongly influence generation quality.

--------------------------------------------------
43.1 Exact callback / hook signatures matter a lot
--------------------------------------------------

One of the easiest ways to break a plugin is to use an almost-correct callback name or wrong argument list.

### Verified strong rule

For send interception, prefer:

```py
def on_send_message_hook(self, account, params):
    ...
```

Do not assume variants like:
- `on_send_message(self, params)`
- `on_message_hook(...)`
- `on_send(...)`

unless the runtime proof explicitly supports them.

AI rule:
- if a hook name is not verified, do not make it the primary integration path
- prefer canonical verified callbacks

--------------------------------------------------
43.2 Prefer verified helpers over reflection-heavy send internals
--------------------------------------------------

In practice, send-related plugins were much more stable when they used verified helpers like:
- `client_utils.send_document(...)`
- other confirmed helper functions

And much less stable when they tried to directly rebuild Telegram’s sending internals via reflection.

Example lesson:
- blindly calling internal `SendMessagesHelper.prepareSending...`-style methods can produce fake success, wrong arg order, or silent non-send
- using verified helper wrappers is much safer

AI rule:
- if a helper exists, use the helper
- only drop into internal reflection if no helper path exists and the user explicitly wants deeper risky behavior

--------------------------------------------------
43.3 PluginSettingsActivity is not always reliable
--------------------------------------------------

Real-world lesson:
- standard ExteraGram settings screens can be unstable depending on:
  - icon names
  - null values
  - selector defaults
  - unexpected settings types
  - plugin cache / stale registration

Observed failure patterns:
- settings open but show garbage / placeholder rows
- settings silently fail to render
- null values crash internal maps
- fancy icons or risky field configs break registration

### Safe settings rules

When using `create_settings()`:
- prefer plain `Header`, `Text`, `Switch`, `Input`, `Selector`
- avoid risky icon usage unless you know the icon exists
- make selector defaults type-correct
- avoid null defaults
- normalize settings before the screen opens

### Very important fallback

If the standard settings screen is unstable:
- build in-plugin settings UI as a custom overlay
- open it from menu/button/command
- keep the plugin usable even if `PluginSettingsActivity` is weak

This is a real, battle-tested design strategy.

--------------------------------------------------
43.4 Normalize settings aggressively
--------------------------------------------------

A recurring source of crashes:
- old settings saved as string
- new code expects int
- selector saved as string label instead of index
- imported preset contains null
- one version stores `"true"` and another expects boolean

AI rule:
- every serious plugin should have a settings normalization layer

Good pattern:

```py
def _ensure_defaults(self):
    for key, value in DEFAULT_PLUGIN_SETTINGS.items():
        current = self.get_setting(key, None)
        if current is None:
            self.set_setting(key, value)
```

And for typed values:
- parse int safely
- clamp values
- map strings to canonical values
- fallback if invalid

--------------------------------------------------
43.5 Java collections are not Python collections
--------------------------------------------------

One of the most common Chaquopy mistakes:
- assuming Java `ArrayList` is directly iterable like a Python list

Real failure:
- `for item in results:` on Java `ArrayList`
- runtime throws `TypeError: 'ArrayList' object is not iterable`

AI rule:
- whenever a value likely comes from Java runtime, do not assume Python sequence semantics

Safer pattern:
- check `size()`
- access with `get(i)`
- create a helper adapter for Java lists

Example mental model:

```py
size = int(java_list.size() or 0)
for i in range(size):
    item = java_list.get(i)
```

This also applies to:
- `document.attributes`
- `reactions.results`
- many Telegram controller arrays

--------------------------------------------------
43.6 Primitive type mismatches can crash hooks instantly
--------------------------------------------------

Real-world failure:
- a hook replaced an `int` argument with Python `int` / Java `Long`
- method expected primitive `int`
- client crashed

Rule:
- when modifying Java method args, respect primitive expectations exactly

If a method expects:
- `int` -> use proper integer-compatible object / safe primitive path
- `long` -> use long-compatible path

Even better:
- avoid modifying args directly if a safer “after original” approach exists

Practical AI rule:
- if a hook crashes due to primitive mismatch, prefer post-action augmentation over argument replacement

--------------------------------------------------
43.7 Recycler/list-row reuse is extremely dangerous
--------------------------------------------------

Real failures in chat-list style plugins:
- rows become randomly tiny
- empty gaps appear
- wrong rows inherit old alpha/scale/translation
- things disappear without user action

Cause:
- Telegram reuses cells
- plugin mutated visual state on the row itself
- row got rebound for another chat/message

Strong rule:
- do not leave `alpha`, `scale`, `translation`, padding, enabled-state or timers on reusable rows
- if you mutate a cell, reset only plugin-owned state
- better use an overlay/progress layer than mutating the row layout itself

This is especially important for:
- `DialogCell`
- `RecyclerListView` rows
- reusable message rows

--------------------------------------------------
43.8 Cancel timers when screen lifecycle changes
--------------------------------------------------

Another real failure:
- long-hold timers survive after navigation
- user opens a chat normally
- delayed action fires later and causes wrong hide/select/etc.

Rule:
- any hold timer / delayed action attached to a screen or row must be cancelled on:
  - `onPause`
  - `onFragmentDestroy`
  - touch cancel / pointer release

This is critical for:
- hidden chats
- drag/hold selection
- delayed menu actions
- prank/troll plugins with delayed triggers

--------------------------------------------------
43.9 Overlay-first design is often better than perfect native injection
--------------------------------------------------

Repeated lesson:
- “perfect native placement” sounds ideal
- but in practice a safe overlay anchored to the correct host often wins

Examples where overlay-first is good:
- profile buttons
- settings panels
- compact tool strips
- backdrop layers
- progress hints

AI rule:
- if native attach is unstable, choose overlay-first
- stable and slightly less native > perfect but broken

--------------------------------------------------
43.10 Root host, parent host, and decorView are different tools
--------------------------------------------------

Do not treat these as interchangeable.

### `fragmentView`
Best for:
- screen-local overlays
- panels that should scroll/clip with the screen

### Parent host of `fragmentView`
Best for:
- elements that need slightly higher positioning than the fragment content
- overlays just outside the screen’s own content layer

### `decorView`
Best for:
- true full-screen backdrops
- activity-wide overlays
- stable top-level visual layers

If a plugin needs:
- full profile backdrop
- global dim / intro animation
- topmost overlay independent of content clipping

Then `decorView` can be the correct choice.

--------------------------------------------------
43.11 Backdrop image logic should be separated from avatar logic
--------------------------------------------------

Real lesson from profile-visual plugins:
- using a profile avatar widget directly as if it were a full-screen backdrop can create weird results
- avatar presentation and backdrop presentation are different jobs

AI rule:
- if image acts like a background, think in terms of:
  - cover mode
  - large host container
  - scrim
  - fade
  - optional blur

Do not assume avatar rendering defaults are good for backgrounds.

--------------------------------------------------
43.12 BackupImageView vs ImageView
--------------------------------------------------

Use `BackupImageView` when:
- the source is Telegram-native image data
- you want Telegram image loading behavior
- you need avatar / profile / chat image integration

Use plain `ImageView` when:
- you already have your own bitmap/file
- you control the source directly
- you just need display, not Telegram media loading semantics

AI rule:
- choose image widget by source semantics, not by habit

--------------------------------------------------
43.13 For large images, think in terms of “cover stack”
--------------------------------------------------

A robust visual stack for profile/chat backdrops:

1. image layer
2. optional blur
3. scrim / dark overlay
4. optional glow / tint
5. content above it

This pattern repeatedly works better than trying to solve readability with alpha only.

--------------------------------------------------
43.14 In-plugin settings overlays are a first-class pattern
--------------------------------------------------

Do not think of in-plugin settings overlays as a hack.
In real ExteraGram plugin work they are often the best solution.

Why:
- independent of plugin settings screen quirks
- can be more beautiful
- can be contextual to current plugin screen
- can expose richer controls

Very strong use-cases:
- file editor plugin
- proxy dashboard
- music card designer
- profile visual plugins
- utility panel plugins

--------------------------------------------------
43.15 When settings screens are ugly, improve hierarchy before adding more color
--------------------------------------------------

Real UI lesson:
- ugly settings are often a structure problem, not a color problem

Improve in this order:
1. sections
2. row density
3. title/subtitle hierarchy
4. value placement
5. accent chips/badges
6. color polish

If values are too long:
- summarize in the row
- open the full text in a dialog/detail view

--------------------------------------------------
43.16 Long values should not live directly in rows
--------------------------------------------------

Real bug:
- supported extensions list was rendered directly inside a row value
- row became enormous
- settings screen looked broken

Rule:
- long datasets in settings rows should show:
  - summary label
  - count
  - short compact preview
  - full text only on tap

Examples:
- `18376 keywords`
- `64 formats`
- `12 favorites`

--------------------------------------------------
43.17 Use commands as fallback entry points for plugin UI
--------------------------------------------------

A very strong real-world pattern:
- even if the plugin has menu items and overlays, also add a command entry

Examples:
- `.openfile settings`
- `.vault`
- `.xsheet`

Why:
- menu hook can fail
- user can forget where UI lives
- command gives deterministic access path

AI rule:
- serious plugins should often have both:
  - visible UI/menu path
  - command fallback path

--------------------------------------------------
43.18 Remote-loader plugins need correct remote object ownership
--------------------------------------------------

In GitHub-loader style plugins, one real failure mode is:
- loader calls payload methods on the loader instance
- payload expects its own instance methods/fields
- runtime crashes with missing attributes

Rule:
- if using remote payload architecture:
  - create a real payload plugin instance
  - call lifecycle methods on that payload instance
  - do not pretend the loader object is the payload object

Also:
- if minimalism matters, direct payload URL can be simpler than manifest indirection
- but manifest is better if you need versioning/hash/metadata

--------------------------------------------------
43.19 Tiny loader architecture: recommended split
--------------------------------------------------

If user wants:
- small `.plugin` in ExteraGram
- main code on GitHub

Recommended split:

### Loader responsibilities
- metadata
- fixed remote URL or manifest URL
- fetch remote code
- cache code locally
- instantiate payload plugin
- forward lifecycle calls

### Payload responsibilities
- all real plugin logic
- settings
- hooks
- menus
- commands

### Optional manifest responsibilities
- version
- hash
- payload URL
- compatibility metadata

--------------------------------------------------
43.20 Public GitHub direct payload is not secret
--------------------------------------------------

Operational lesson:
- if loader downloads payload from public GitHub, users can find the payload
- even private GitHub is not secure if credentials/token must be shipped in the plugin

AI rule:
- do not describe public raw GitHub as secret distribution
- if secrecy is required, a server/proxy is needed

--------------------------------------------------
43.21 Debug-friendly plugins are much easier to stabilize
--------------------------------------------------

When a feature is risky:
- log class availability
- log attach success/failure
- log key branch decisions
- log callback start/end

Good logging pattern:
- short prefix
- no spam in hot loops
- enough context to identify the failing stage

Examples:
- hook attached?
- fragment found?
- file path resolved?
- Java list size?
- send helper returned?

--------------------------------------------------
43.22 Real plugin generation should use “stability ladder”
--------------------------------------------------

For every requested feature:

1. verified helper path
2. lifecycle-safe overlay path
3. local class probe path
4. internal reflection path
5. abandon/limit feature if still unverified

This should be the default AI decision ladder.

==================================================
46. DANGEROUS ANTI-PATTERNS TO AVOID
==================================================

These are high-signal anti-patterns extracted from actual plugin failures.

Do not:
- assume Java `ArrayList` is Python-iterable
- assume selector/default types are automatically normalized
- inject giant raw lists directly into a settings row
- modify reusable list row transforms and forget to reset them
- keep delayed hold timers alive after navigation
- replace primitive Java args with wrong boxed types
- use internal send reflection when verified helper exists
- attach deep native UI into an unstable screen host just for perfection
- assume a GitHub remote payload is private if the loader can fetch it publicly
- assume upstream Telegram source names are runtime proof
- use standard settings screen as the only control surface for complex plugins

class AttachHook(MethodHook):
    def __init__(self, plugin):
        self.plugin = plugin

    def after_hooked_method(self, param):
        self.plugin._schedule_attach(param.thisObject)


class DestroyHook(MethodHook):
    def __init__(self, plugin):
        self.plugin = plugin

    def before_hooked_method(self, param):

class Plugin(BasePlugin):

        self._attached = {}

    if values is None:
        return
    try:
        size = int(values.size() or 0)

        return

48.9 Remote loader skeleton
--------------------------------------------------

Use when:
- user explicitly wants a tiny plugin file
- payload lives on GitHub

```py
__id__ = "remote_loader"
__name__ = "Remote Loader"

PAYLOAD_URL = "https://raw.githubusercontent.com/user/repo/main/payload.py"

import types
import urllib.request
from base_plugin import BasePlugin


class Plugin(BasePlugin):
    def __init__(self):
        super().__init__()
        self._payload = None

    def _load_payload(self):
        if self._payload is not None:
            return self._payload
        code = urllib.request.urlopen(PAYLOAD_URL, timeout=10).read().decode("utf-8")
        module = types.ModuleType("remote_payload")
        exec(compile(code, "remote_payload.py", "exec"), module.__dict__)
        payload_cls = getattr(module, "Plugin", None)
        if payload_cls is None:
            raise RuntimeError("Remote payload has no Plugin class")
        self._payload = payload_cls()
        return self._payload
```
